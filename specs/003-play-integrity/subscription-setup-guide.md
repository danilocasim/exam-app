## 1. Play Console Setup

### Prerequisites

- Google Play Developer account ($25 one-time registration)
- App published to at least **internal test track** (subscriptions cannot be created for draft apps)
- Google Cloud project with **Google Play Android Developer API** enabled (for server-side validation)
- Service account with `androidpublisher` scope (reuses Play Integrity service account)

### Steps

1. **Open Google Play Console** → Select (or create) your app
2. Navigate to **Monetize → Products → Subscriptions**
3. Create **one subscription group** per app (e.g., "Dojo Exam CLF-C02 Premium")
4. Within the group, create **3 subscription products** (monthly, quarterly, annual)
5. For each product, create a **base plan** with the appropriate billing period and price
6. Activate all products

## 2. Subscription Product ID Convention

Product IDs follow a strict naming convention used by both the mobile app and backend:

```
{plan}_{exam_type_id_lowercased_underscored}
```

### Format Rules

- **plan**: `monthly`, `quarterly`, or `annual`
- **exam_type_id**: The exam type ID with hyphens replaced by underscores, lowercased

### Examples

| Exam Type | Plan | Product ID |
|-----------|------|------------|
| CLF-C02 | Monthly | `monthly_clf_c02` |
| CLF-C02 | Quarterly | `quarterly_clf_c02` |
| CLF-C02 | Annual | `annual_clf_c02` |
| SAA-C03 | Monthly | `monthly_saa_c03` |
| SAA-C03 | Quarterly | `quarterly_saa_c03` |
| SAA-C03 | Annual | `annual_saa_c03` |

### Code Reference

The SKU is generated by `getSubscriptionSku()` in `packages/shared/src/services/billing.service.ts`:

```typescript
const normalizedId = examTypeId.toLowerCase().replace(/-/g, '_');
return `${plan}_${normalizedId}`;
```

Each app's config in `apps/{app}/src/config/app.config.ts` also declares `subscriptionSkus` that must match these Play Console product IDs.

## 3. Pricing Table

| Plan | Price (USD) | Billing Period | Savings |
|------|------------|----------------|---------|
| Monthly | $2.99 | 1 month (P1M) | — |
| Quarterly | $6.99 | 3 months (P3M) | ~22% vs monthly |
| Annual | $19.99 | 12 months (P1Y) | ~44% vs monthly |

### Notes

- Quarterly is the **recommended** plan (best value proposition for users)
- Prices are set per-country in Play Console; the above are USD base prices
- Google takes a 15% commission on subscriptions (after the first year at 30%)

## 4. Creating Subscription Products

For **each exam app** (e.g., CLF-C02, SAA-C03), create 3 products:

### Step-by-Step (per product)

1. Go to **Monetize → Subscriptions → Create subscription**
2. **Product ID**: Enter the ID following the convention (e.g., `quarterly_clf_c02`)
   - ⚠️ Product IDs are **permanent** — they cannot be changed after creation
3. **Name**: e.g., "Quarterly Premium Access"
4. Click **Save**

### Create Base Plan

1. Within the subscription, click **Add base plan**
2. **Base plan ID**: Use `base` (or leave the default)
3. **Auto-renewing**: Yes
4. **Billing period**: Select the correct period:
   - Monthly → 1 month
   - Quarterly → 3 months
   - Annual → 1 year
5. **Price**: Set the regional pricing
   - Click "Set price" → Enter $2.99 / $6.99 / $19.99
   - Use "Auto-convert" for other currencies
6. **Grace period**: 7 days (recommended — gives users time to fix payment issues)
7. **Account hold**: 30 days (recommended)
8. **Resubscribe**: Allow (users can resubscribe from Play Store)
9. Click **Save** → **Activate**

### Repeat for All Plans

Create all 3 products × each exam app. For a two-app setup (CLF-C02, SAA-C03), that's 6 total subscription products.

## 5. Testing with License Testers

License testers can make test purchases without being charged.

### Setup

1. Go to **Play Console → Settings → License testing**
2. Add tester email addresses (Gmail accounts)
3. Set **License response** to: `RESPOND_NORMALLY`

### Test Subscription Behavior

License testers get **accelerated subscription renewals** for testing:

| Real Duration | Test Duration |
|---------------|---------------|
| 1 week | 5 minutes |
| 1 month | 5 minutes |
| 3 months | 10 minutes |
| 6 months | 15 minutes |
| 1 year | 30 minutes |

- Subscriptions renew up to **6 times** before auto-cancelling
- Grace period: 5 minutes (instead of days)
- Account hold: 10 minutes (instead of days)

### Important Notes

- Tester must be signed into the **primary Google account** on the device
- The app must be installed from **Play Store** (internal/closed/open test track)
- Test subscriptions are automatically cancelled after 6 renewals
- Clear Play Store cache if test purchases don't appear

## 6. Testing with Internal Test Track

The internal test track allows up to 100 testers without review.

### Upload to Internal Test Track

1. Build the release APK/AAB:
   ```bash
   cd apps/aws-clp && eas build --profile production --platform android
   ```
2. Go to **Play Console → Testing → Internal testing**
3. **Create a new release** → Upload the AAB
4. **Add testers**: Create an email list with tester addresses
5. **Save and roll out**

### Install on Device

1. Testers will receive an **opt-in link** (share via Play Console)
2. Tester opens the link on their device → accepts → installs from Play Store
3. Alternatively: direct install link format:
   ```
   https://play.google.com/apps/internaltest/{appId}
   ```

### Test Flow

1. Open app → navigate to **Upgrade** screen
2. Select a plan (e.g., Quarterly)
3. Complete purchase via Play Store dialog (no charge for license testers)
4. Verify: app unlocks full question bank (PREMIUM tier)
5. Wait for test expiry → verify auto-downgrade to FREE
6. Tap "Restore Subscription" → verify restoration works

## 7. Server-Side Validation (T262.5)

The backend provides a `POST /api/billing/verify-subscription` endpoint for server-side subscription verification.

### Endpoint

```
POST /api/billing/verify-subscription
Authorization: Bearer <jwt_token>
Content-Type: application/json
```

### Request Body

```json
{
  "productId": "quarterly_clf_c02",
  "purchaseToken": "<token_from_play_store>",
  "packageName": "com.danilocasim.dojoexam.clfc02"
}
```

### Response

```json
{
  "valid": true,
  "expiryTimeMillis": 1735689600000,
  "autoRenewing": true,
  "paymentState": 1,
  "cancelReason": null
}
```

### Payment State Values

| Value | Meaning |
|-------|---------|
| 0 | Payment pending / on hold / paused / expired |
| 1 | Payment received (active, grace period, or cancelled but not yet expired) |

### Cancel Reason Values

| Value | Meaning |
|-------|---------|
| 0 | User cancelled |
| 1 | System cancelled (billing failure) |
| 2 | Developer cancelled (revoked) |
| 3 | Replaced by another subscription |

### Service Account Setup

The billing verification reuses the same service account as Play Integrity:

1. **Google Cloud Console** → IAM & Admin → Service Accounts
2. Ensure the service account has **Android Publisher** permissions
3. In Play Console: **Settings → API access** → Link the Google Cloud project
4. Grant the service account **"View financial data, orders, and cancellation survey responses"** permission

### Environment Variables

```env
# Inline JSON (recommended for Railway/production)
GOOGLE_SERVICE_ACCOUNT_KEY={"type":"service_account","project_id":"...","private_key":"..."}

# OR file path (local development)
GOOGLE_SERVICE_ACCOUNT_KEY_PATH=keys/service-account.json
```

## 8. Subscription Lifecycle

### State Diagram

```
┌──────────┐     Purchase     ┌────────┐
│   FREE   │ ───────────────► │ ACTIVE │
└──────────┘                  └────┬───┘
     ▲                             │
     │                    Payment fails
     │                             │
     │                             ▼
     │                    ┌──────────────┐    Payment fixed    ┌────────┐
     │                    │ GRACE PERIOD │ ──────────────────► │ ACTIVE │
     │                    │   (7 days)   │                     └────────┘
     │                    └──────┬───────┘
     │                           │
     │                  Still not paid
     │                           │
     │                           ▼
     │                    ┌──────────────┐    Payment fixed    ┌────────┐
     │                    │ ACCOUNT HOLD │ ──────────────────► │ ACTIVE │
     │                    │  (30 days)   │                     └────────┘
     │                    └──────┬───────┘
     │                           │
     │               Not paid within hold
     │                           │
     │                           ▼
     │    ┌───────────┐    ┌───────────┐
     └────│  EXPIRED  │◄───│ CANCELLED │
          └───────────┘    └───────────┘
               ▲                 ▲
               │                 │
          Hold expires     User cancels
                          (access until
                           expiry date)
```

### State Descriptions

| State | User Access | Description |
|-------|-------------|-------------|
| **Active** | ✅ Full access | Subscription is paid and current |
| **Grace Period** | ✅ Full access | Payment failed; user retains access for 7 days while Google retries |
| **Account Hold** | ❌ No access | Payment still failing after grace period; access revoked for 30 days |
| **Cancelled** | ✅ Until expiry | User cancelled but subscription runs until end of billing period |
| **Expired** | ❌ No access | Subscription ended; user downgraded to FREE tier |
| **Paused** | ❌ No access | User paused subscription (if enabled); resumes automatically |

### App Behavior per State

- **Active / Grace Period**: `tierLevel = 'PREMIUM'`, full question bank access
- **Cancelled (not yet expired)**: `tierLevel = 'PREMIUM'`, show "Subscription ending" banner
- **Account Hold**: `tierLevel = 'FREE'`, show "Payment issue" banner with link to fix
- **Expired**: `tierLevel = 'FREE'`, show upgrade screen to resubscribe

### Lifecycle at App Launch

The app runs `checkExpiryAndRestore()` on each launch:

1. If `expiryDate` is in the future → keep PREMIUM (no action)
2. If `expiryDate` is past + `autoRenewing = false` → downgrade to FREE
3. If `expiryDate` is past + `autoRenewing = true` → call `restoreSubscription()`:
   - If renewed purchase found → update expiry, stay PREMIUM
   - If no renewal found → downgrade to FREE

## 9. Troubleshooting

### Common Issues

#### "This item is not available in your country"
- The pricing for the user's country has not been set
- Solution: In Play Console, set prices for all countries or use "Auto-convert" from USD base price

#### "Error: BillingClient is not ready"
- `initBilling()` was not called before attempting a purchase
- The app is a debug build not installed from Play Store
- Solution: Use internal test track for testing billing

#### "Purchase token invalid" (server-side validation)
- Token has expired (tokens are only valid for a limited time after expiry)
- Solution: Use the most recent token from `getAvailablePurchases()`

#### "E_USER_CANCELLED" error
- Normal behavior when user dismisses the purchase dialog
- The app handles this gracefully (returns `null` from `processPurchaseError`)

#### Purchase stuck in PENDING state
- Some payment methods (carrier billing, convenience store payments) have delayed processing
- Solution: The app tracks pending state via `pendingProductId` in the purchase store and checks on next launch via `checkPendingSubscriptions()`

#### Subscription not restoring on new device
- Ensure the user is signed in with the **same Google account** that made the purchase
- Call `restoreSubscription()` after Google Sign-In completes
- Check that the Play Store app is up to date on the device

#### "BILLING_UNAVAILABLE" error
- Play Store app is outdated or Google Play Services is unavailable
- Device was sideloaded (no Play Store access)
- Solution: The `UpgradeScreen` shows a "Play Store unavailable" error banner

#### Unacknowledged purchases being refunded
- Google refunds unacknowledged purchases after 3 days
- Solution: The app calls `acknowledgePurchase()` immediately after a successful purchase and also re-acknowledges unacknowledged purchases on each app launch via `checkPendingSubscriptions()`

### Debug Logging

In development mode (`__DEV__ = true`), all billing operations log to console with `[Billing]` prefix:

```
[Billing] Mock subscription purchase in dev mode: quarterly_clf_c02
[Billing] Subscription purchase complete: { productId: 'quarterly_clf_c02', plan: 'quarterly', ... }
```

In production, key events are logged:
- Connection initialization
- Purchase attempts and results
- Acknowledgement success/failure
- Expiry checks and restoration attempts

## 10. Play Console Reporting & Analytics

### Subscription Dashboard

Navigate to **Play Console → Monetize → Financial reports → Subscriptions** to view:

- **Active subscriptions**: Count of currently active subscribers
- **New subscriptions**: Subscriptions acquired in a time period
- **Cancellations**: Voluntary and involuntary (payment failure) cancellations
- **Revenue**: Gross and net revenue after Google's commission

### Key Metrics to Monitor

| Metric | What to Watch |
|--------|---------------|
| **Conversion rate** | % of FREE users who subscribe (track per plan) |
| **Churn rate** | % of subscribers who cancel per period |
| **Grace period recovery** | % of users who fix payment during grace period |
| **Plan distribution** | Which plan (monthly/quarterly/annual) is most popular |
| **Revenue per user** | Average revenue per subscribing user |
| **Retention** | How long subscribers remain active |

### Real-Time Developer Notifications (RTDN)

For advanced tracking, set up [Real-Time Developer Notifications](https://developer.android.com/google/play/billing/getting-ready#configure-rtdn):

1. Create a **Cloud Pub/Sub topic** in Google Cloud
2. In Play Console: **Monetize → Monetization setup → Real-time developer notifications**
3. Enter the Pub/Sub topic name
4. Subscribe to the topic and process events server-side

RTDN delivers immediate notifications for:
- New purchases
- Renewals
- Cancellations
- Holds and pauses
- Grace period entries
- Revocations

> **Note**: RTDN is optional for the current implementation. The app relies on client-side purchase listeners and periodic `checkExpiryAndRestore()` calls. RTDN would be beneficial for real-time analytics and server-side subscription state tracking at scale.

### Google Play Console API

For programmatic access to subscription data, use the [Google Play Developer API](https://developers.google.com/android-publisher):

- `purchases.subscriptionsv2.get` — Query individual subscription status (used by T262.5)
- `monetization.subscriptions.list` — List subscription products
- `reviews.list` — Monitor user reviews related to billing issues
